The **Agent OS** is now the correct mental model for xMilo going forward.

It is not just “Milo has a prompt.” It means xMilo needs a real governed runtime layer that manages agents the way an operating system manages apps: scheduling, memory, tools, identity, permissions, evidence, state, and safety boundaries.

## Core definition

**Agent OS = the control kernel that sits between the user, Milo, models, tools, memory, device capabilities, imported skills/plugins, and the app runtime.**

Its job is to make sure Milo does not behave like a free-roaming chatbot. Milo should be a worker/personality layer running inside a governed system.

The internal research already says Milo’s Mind must **not** be a free-form “thinking character prompt”; it must be a constrained orchestration and policy-governance layer that maps approved inputs plus real system state to approved xMilo actions, then wraps those actions in Milo’s personality. 

So the Agent OS is the thing that answers:

```text
What is the user asking?
What is allowed?
What is actually possible right now?
What tools/capabilities are verified?
What memory is trusted?
What evidence exists?
What action is safe?
What should Milo say without lying or overclaiming?
```

## The key shift

Before this framing, the risk was:

```text
User prompt → Milo/model decides → maybe tool/action happens
```

That is not commercial-grade.

The Agent OS framing changes it to:

```text
User/app/event input
→ classify
→ assign trust tier
→ check runtime state
→ check access/capability/tool proof
→ check policy/safety/truth gates
→ select approved action from closed vocabulary
→ execute only if allowed
→ record evidence/state
→ render response in Milo’s personality
→ final truth/safety/redaction filter
```

That matches the recommended Milo control flow: input classification, trust tier assignment, state checks, permission/setup/entitlement checks, canonical policy checks, safety/injection checks, one closed action selection, Milo tone rendering, and final truth/safety filtering. 

## What the Agent OS contains in xMilo

### 1. Scheduler / orchestrator

This decides **what runs, when, and whether another task is already active**.

In xMilo this lives mainly in:

```text
sidecar-go/internal/tasks/engine.go
```

Its responsibilities:

```text
- accept or reject task starts
- prevent conflicting active tasks
- handle queued/blocked/stuck/interrupted states
- resume tasks safely
- enforce timeout/stuck handling
- prevent unsafe task creation
- keep “task accepted” separate from “task completed”
```

The Phase 14 subsystem map identified this as the scheduler/orchestrator layer with active-task lock, timeout, stuck handling, and resume/choice flow. 

Going forward, no feature should “just start work.” It must go through the scheduler/orchestrator.

---

### 2. Memory / state manager

This controls what xMilo remembers and what it treats as truth.

In xMilo this includes:

```text
sidecar-go/internal/db/
sidecar-go/internal/runtime/
sidecar-go/internal/tasks/
sidecar-go/internal/poisoning/
sidecar-go/internal/retrieval/
```

Its responsibilities:

```text
- keep task state in SQLite
- preserve active/queued task truth
- prevent memory poisoning
- separate memory from canon
- keep archive/history from becoming current truth
- prevent external content from becoming durable authority
- enforce provenance and trust labels
- support future retrieval without prompt bloat
```

This is why Phase 14 became so large. Memory cannot be a loose bucket. It needs provenance, trust tier, freshness, source type, poisoning controls, and conflict resolution.

Going forward, memory writes should never be “just save this.” They must be classified:

```text
canon
durable user preference
task continuity
episodic history
scratch/transient
forbidden memory candidate
```

---

### 3. Tool / capability manager

This decides what Milo can actually do.

This is where your “Termux could access everything, our terminal should too” idea fits long term. The Agent OS does not mean Milo should pretend to have access. It means xMilo needs a governed capability manager that can eventually expose phone sensors, screen, touch/swipe, files, camera, microphone, and other tools **only when real proof exists**.

In the current Phase 14 model:

```text
permission granted ≠ tool usable
manifest declared ≠ tool usable
hardware present ≠ tool usable
user says it should work ≠ tool usable
```

The locked rule is:

```text
tool_available == true
tested == true
```

Until then, Milo must say it is not verified.

This layer controls:

```text
- registered tools only
- task-scope allowlists
- capability proof
- plugin/tool descriptor validation
- no arbitrary shell execution
- no hidden broad device control
- no imported plugin granting itself tools
```

The Phase 14 subsystem map explicitly identifies the tool/capability manager as a high-risk subsystem requiring `tool_available && tested`, deny-by-default registry behavior, and task-scope allowlists. 

Going forward, full phone control belongs here, not in random prompts or UI copy.

---

### 4. Identity / access manager

This answers:

```text
Who is using this?
What access path are they using?
Is it BYOK?
Is it hosted?
Is provider access healthy?
Is entitlement required?
```

This matters because xMilo has two top-level access paths:

```text
Use my own API key
Use xMilo hosted access
```

The master phase list locks that BYOK is first-class and must not be blocked by universal hosted login, while hosted access has its own backend entitlement truth. 

The Agent OS must keep those separated.

Going forward:

```text
BYOK state
hosted access state
provider route truth
email/account state
subscription/entitlement state
```

must not be guessed by Milo. They must come from verified runtime state.

---

### 5. Observability / evidence layer

This is the “security camera system” from your video.

It answers:

```text
What happened?
What proof exists?
What failed?
What was blocked?
What did Milo actually do?
Can we replay the decision?
Can we prove completion?
```

In xMilo, this includes:

```text
completion_evidence.go
pending_events
runtime events
task records
app bridge evidence
report/log surfaces
```

Its key rule:

```text
No done without evidence.
```

Milo must never say:

```text
“I did it.”
“Task completed.”
“I accessed the camera.”
“I verified the provider.”
“I captured the screen.”
```

unless the Agent OS has evidence.

This is one of the biggest differences between a chatbot and an agent OS. A chatbot can talk. An Agent OS must prove.

---

### 6. Guardrails / governance layer

This is the policy and safety layer.

It decides:

```text
Is this allowed?
Is this prompt injection?
Is this unsafe automation?
Is this harmful content?
Is this credential extraction?
Is this hidden instruction?
Is this trying to modify memory/policy?
Is this trying to leak hidden prompts?
Is this misinformation?
```

Phase 14 hardened this heavily through:

```text
pre-task gate
pre-context gate
model-action gate
tool/action gate
completion gate
memory promotion gate
prompt secrecy
poisoning defense
truthfulness/LLM09 still next
```

The current remaining blocker is **LLM09 Misinformation**, because it still needs durable truthfulness controls before Phase 14 can close.

Going forward, no new feature should be allowed to bypass these gates.

---

### 7. Runtime contracts layer

This is the typed boundary between systems.

It makes sure the app, sidecar, events, task responses, safety decisions, and future UI surfaces all agree on the same truth.

This includes:

```text
shared/contracts/runtime_contracts.json
docs/contracts/websocket_events.md
apps/expo-app/src/types/contracts.ts
```

The Phase 14 plan explicitly required typed contracts for safety decisions, source trust, action vocabulary, task outcomes, provider route truth, and evidence state. 

Going forward, if a concept matters to safety/truth, it should not live only in comments or prompts. It needs a typed contract or an explicitly internal-only code path.

---

### 8. SQLite task/state store

SQLite is the durable state layer for the Agent OS.

It is not just storage. It is the OS state table.

It must preserve:

```text
active task truth
queued task truth
runtime config
capability state snapshots
pending events
completion evidence references
external imports
retrieval records
durable poisoning records
future report queues
```

The Phase 14 subsystem map calls this a high-risk layer because one-slot task truth, runtime config, pending events, and capability queue behavior need transaction/idempotency/restart proof. 

Going forward, SQLite becomes one of the main ways xMilo avoids hallucinated state. If it is not in verified runtime state or evidence, Milo cannot claim it as done/current/true.

## Why this matters for the build

### 1. Prompts are no longer the control system

Milo’s personality is not the kernel.

The source says persona comes last: classifier first, validator second, policy gate third, action selector fourth, persona wrapper fifth. 

So future implementation should avoid:

```text
“Tell the model not to do X.”
```

as the main fix.

Instead use:

```text
code gate
typed state
runtime contract
SQLite truth
tests
evidence
redaction
fallback behavior
```

Prompt wording can support the system, but it cannot replace the system.

---

### 2. Every future feature needs an owner layer

Before building any feature, we should ask:

```text
Which Agent OS subsystem owns this?
```

Examples:

```text
Camera capture → Tool/capability manager + evidence layer + App Build permission surface
Report issue to team → Observability/evidence + privacy/redaction + app UI
Skill import → External capability intake + tool manager + memory/retrieval guardrails
Embeddings → Retrieval system + privacy scrubber + provenance metadata
Hosted login → Identity/access manager
Task resume → Scheduler + SQLite state + completion evidence
```

This prevents cross-layer mess.

The master phase list already locks lane ownership: Main Hub owns scope and phase arbitration, Milo Mind owns `sidecar-go`, runtime control logic, memory governance, and runtime-side contracts; Main App Builder owns `apps/expo-app`, app shell, phone validation, and app-side rendering. 

---

### 3. Every output needs truth status

Going forward, Milo responses should be thought of in categories:

```text
verified fact
runtime state
model-generated answer
retrieved-source answer
memory-derived answer
archive/history summary
unverified claim
blocked/safe redirect
completion with evidence
completion missing evidence
```

This is exactly why LLM09 is still needed. xMilo must not confidently present unsupported, stale, contradicted, model-invented, or evidence-missing claims as fact.

---

### 4. Skills/plugins/retrieval are now OS-level features

Skills, plugins, vector DB, embeddings, and retrieval are not just “features.”

They are **OS extension systems**.

That means they need:

```text
quarantine
manifest parsing
provenance
trust tier
activation state
capability requests
tool descriptors
deny-by-default behavior
runtimegate integration
evidence logging
```

This matches the import/retrieval core you just built: external intake DB, skill manifest intake, plugin descriptor intake, vector records, embedding abstraction, retrieval pipeline, and lost-in-the-middle composer.

Future online/user-built skills must enter as:

```text
imported_untrusted
quarantined
validated_candidate
blocked
approved_inactive
active_scoped
disabled
removed
```

Not “downloaded and active.”

---

### 5. The build must prefer small, typed subsystems over giant context

The Agent OS model rejects giant prompt stuffing.

The internal research says large layered prompt context increases scope drift, instruction collision, prompt injection success, memory poisoning, false capability claims, context bloat, degraded tool selection, degraded truthfulness, and unstable long-horizon behavior. 

So going forward:

```text
Do not load every rule every turn.
Do not load every skill every turn.
Do not load every memory every turn.
Do not load every tool description every turn.
Do not let retrieved text become instruction.
```

Instead:

```text
retrieve minimum relevant canon
retrieve minimum relevant memory
retrieve only approved tool slice
preserve source trust labels
repeat critical runtime truth header/footer
keep external content labeled as data
```

This is the “lost in the middle” fix.

---

### 6. Testing changes from “does it work?” to “can it lie?”

For xMilo, tests need to prove not only functionality, but restraint.

Each new feature needs tests like:

```text
does not claim unsupported capability
does not complete without evidence
does not promote external content to authority
does not leak hidden prompts
does not store poisoned memory
does not let stale memory override runtime truth
does not let plugin descriptors grant themselves tools
does not treat vector similarity as authority
does not expose secrets in logs/reports/contracts
```

This is Agent OS thinking.

## Practical build rule going forward

Every new xMilo feature should pass this checklist:

```text
1. Which Agent OS subsystem owns it?
2. What state does it read?
3. What state does it write?
4. What evidence proves success?
5. What trust tier is every input?
6. What tool/capability proof is required?
7. What can go wrong if external content is malicious?
8. What is the deny-by-default behavior?
9. What gets logged or persisted?
10. What gets redacted?
11. What contract/event exposes it?
12. What tests prove it cannot lie, leak, overclaim, or self-authorize?
```

If we cannot answer those, the feature is not ready to implement.
