PURPOSE
This file is the canonical startup/setup flow for xMilo from Play Store download through entry into the live Lair.
It exists to stop startup-flow drift.
It defines:
- exact card order
- exact card names
- exact branch points
- what belongs in startup
- what must be moved to Settings
- how command cards must behave
- when the 12-hour timer starts
- what should happen before the user reaches the Lair

CORE STARTUP RULES
1. Startup must use one active overlay card at a time.
2. Startup must keep the live Lair/map behind the cards.
3. No giant combined setup slab.
4. No multi-action checklist card containing many unrelated buttons at once.
5. Every active card must represent the smallest current required next action.
6. Educational explanation must precede any sensitive permission or external handoff.
7. If a user is being sent to another app or settings surface, the card must explicitly say what app/screen will open and what the user should do there.
8. After leaving the app for a setup action, the user should return to xMilo and continue or re-check from a dedicated return card.
9. Advanced runtime/debug/reference pages do not belong in first-run startup unless Main Hub explicitly marks them startup-critical.
10. Startup must remain Play-safe and must not imply hidden installs, background sideloading, or deceptive external linking.

CANONICAL FLOW FROM DOWNLOAD TO LAIR

STEP 1
Name: `Download xMilo`
Type: startup step
What happens:
- User downloads xMilo from the Play Store.

STEP 2
Name: `Open xMilo`
Type: startup step
What happens:
- User opens xMilo for the first time.

STEP 3
Name: `xMilo startup overlay`
Type: shell state
What happens:
- xMilo opens into the live Lair-backed onboarding shell.
- Onboarding appears as floating cards over the live Lair.
- One card at a time.

STEP 4

STEP GROUP 3 — X MILO PERMISSION GATE

RULE FOR THIS GROUP
- Permission cards come before Termux workshop command cards when those permissions are needed first.
- Each permission-related step is its own card when needed.
- Do not dump all permissions into one startup page.

STEP 14
Name: `Allow xMilo permissions`
Type: explanation card
Purpose:
- Explain the exact permission or permission-like access needed for the current step and what breaks if declined.

STEP 15
Name: `Open xMilo permission gate`
Type: single-action card
Purpose:
- Open the exact xMilo permission/settings surface needed for the current step.
Required copy behavior:
- Must say what screen will open.
- Must say exactly what to allow/change there.
- User returns to xMilo afterward.
Primary button:
- `Open xMilo permission gate`

STEP 16
Name: `Permission granted`
Type: return/check card
Purpose:
- Confirm the permission step completed and continue forward.

EXPLICIT PERMISSION / BRIDGE FACTS TO PRESERVE
- The bridge gate is not satisfied by Android app-info screenshots alone.
- The real app-owned gate depends on Termux run-command capability and callback success.
- If Android prompts for Termux run-command permission, xMilo must guide the user through that permission clearly.

STEP GROUP 4 — APP-OWNED BOOTSTRAP AND BRIDGE VERIFICATION

STEP 17
Name: `Prepare xMilo`
Type: bootstrap step
Purpose:
- Run the app-owned bootstrap/preparation stage.

STEP 18
Name: `Verify xMilo connection`
Type: verification step
Purpose:
- Run the explicit end-to-end app-owned bridge verification.
- Package presence, permission presence, API presence, or artifact presence alone are insufficient.

STEP GROUP 5 — GUIDED TERMUX HANDOFF / WORKSHOP

RULE FOR THIS GROUP
- `Guided handoff into Termux` is an intro card only.
- `Open Termux workshop` begins individually named workshop command cards.
- Prefer direct user-initiated Termux action from the button if that is available and compliant.
- Manual copy/paste is fallback, not first choice, when the direct handoff works.
- The successful direct handoff pattern already observed in the current setup should be treated as the preferred behavior target for all similar workshop codes if compliant.

STEP 19
Name: `Guided handoff into Termux`
Type: explanation-only card
Purpose:
- Explain:
  - this will open the Termux app
  - xMilo may run or ask the user to run a setup command
  - after that, the user should return to xMilo and continue
- This card must not be bundled with advanced artifact/release/debug buttons.

STEP 20
Name: `Open Termux workshop`
Type: single-action card
Purpose:
- Open Termux and begin the workshop flow.
Required copy behavior:
- Must explicitly tell the user this will open the Termux app.
- Must tell the user to do nothing else and return to xMilo when the command finishes or when instructed.
Primary button:
- `Open Termux workshop`

STEP 21
Name: `Enable Termux external commands`
Type: workshop command card
Purpose:
- Enable external app command execution for the verified bridge check.
Preferred behavior:
- Button directly sends the command into Termux and runs it from the user-initiated handoff.
Fallback behavior only if necessary:
- `Copy command`
- `Open Termux`
Exact command:
- `mkdir -p ~/.termux`
- `echo "allow-external-apps=true" >> ~/.termux/termux.properties`
- `termux-reload-settings`

STEP 22
Name: `Wake lock for first install`
Type: workshop command card
Purpose:
- Ensure the sidecar/bootstrap path has a better chance of staying alive during first install/runtime setup if this is actually required by the current bootstrap flow.
Preferred behavior:
- Direct user-initiated Termux action from the button if possible.
Fallback:
- `Copy command`
- `Open Termux`
Exact command:
- `termux-wake-lock`

STEP 23
Name: `Run setup spell in Termux`
Type: manual fallback workshop card
Purpose:
- Manual fallback only if automatic Termux handoff is unavailable or fails.
- This is not the normal happy path.
Structure:
- explanation card
- then action card
Preferred behavior:
- direct user-initiated Termux action if possible
Fallback:
- `Copy setup spell`
- `Open Termux`

STEP 24
Name: `Re-check`
Type: verification return card
Purpose:
- User returns to xMilo and taps `Re-check`.
- If the bridge now passes, xMilo continues automatically.
- If it does not pass, xMilo stays on the same narrow guided blocker path.

ADDITIONAL COMMAND-CHECK RULE

STEP 29

Type: skip branch
Purpose:

- The 12-hour timer does not start on:
  - download
  - first open
  - trust pages
  - install cards
  - permission cards
  - bootstrap cards
  - workshop cards
