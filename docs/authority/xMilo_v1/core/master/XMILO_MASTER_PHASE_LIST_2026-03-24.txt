PHASE 1B: Topic-Aware Conversation Lanes (PLANNING LOCK)
- Core definition / purpose:
  - Topic-aware conversation lanes are continuity containers for user-facing conversation context.
  - Their purpose is to preserve clarity and continuity across meaningfully different user topics without turning xMilo into a chat-thread maze.

- User topic lane vs internal lane rule:
  - Topic-aware conversation lanes are user-facing or user-context continuity containers only.
  - Topic-aware conversation lanes are not internal work lanes.
  - Topic-aware conversation lanes are not implementation lanes.
  - Topic-aware conversation lanes are not orchestration lanes.
  - Topic-aware conversation lanes are not verification lanes.
  - Topic-aware conversation lanes are not background task lanes.
  - Existing internal lane concepts and ownership models remain separate and must not be merged into user topic-lane behavior.

- Internal/external separation rule:
  - Internal lane handling remains an implementation and orchestration concern.
  - User topic lanes, if surfaced at all, exist only to preserve user conversation continuity and reduce context confusion.
  - Internal routing, worker assignment, verification state, and execution ownership must not be exposed to the user as topic-lane structure unless explicitly approved elsewhere.
  - If user-visible topic-lane affordances exist, they must be limited, calm, and continuity-oriented rather than operational or technical.

- Topic boundary rule:
  - `same-topic continuation`
    - A follow-up that keeps the same primary user goal, artifact, decision thread, or active problem space.
    - Default behavior: remain in the same topic lane.
  - `subtopic within same lane`
    - A narrower branch, clarification, example, or related follow-up that still materially belongs to the same primary goal or workstream.
    - Default behavior: remain in the same topic lane.
  - `truly new topic`
    - A shift to a meaningfully different primary goal, different problem domain, different long-lived task area, or different continuity context such that keeping the old lane would reduce clarity.
    - Only this class is eligible for a new topic lane.
  - Minor wording changes, small tangents, clarifying questions, shallow side branches, and closely related continuations do not justify a new topic lane by themselves.

- No thread explosion rule:
  - Do not create a new topic lane for every minor follow-up.
  - Do not create a new topic lane for wording shifts alone.
  - Do not create a new topic lane for closely related continuations.
  - Prefer continuity by default.
  - Create a new topic lane only when a meaningful topic boundary is crossed and continuity would otherwise become misleading or cluttered.

- User-visible complexity rule:
  - Topic-aware conversation lanes must not turn the product into a manual thread-management system unless a separate explicit design decision approves that behavior.
  - If topic lanes are user-visible, the interface must prevent clutter, duplication, and maze-like switching behavior.
  - The user must not be forced to actively manage a large set of parallel lanes for normal use.
  - The default user experience must remain simple, continuity-friendly, and low-friction.

- Lane naming / surfacing rule:
  - User topic lane names, summaries, and switching affordances, if surfaced, must describe user topic continuity only.
  - They must not expose internal worker, routing, orchestration, or verification semantics.
  - No weaker lane may reuse internal lane terminology as user topic-lane terminology.

- Conflict-prevention rule:
  - User topic lanes must not silently conflict with existing system lane concepts.
  - Internal work lanes continue to govern ownership, execution, verification, and orchestration.
  - User topic lanes continue to govern continuity of user-facing conversation context only.
  - If a design decision would blur those boundaries, it must be escalated before implementation.

- Proliferation-prevention acceptance rule:
  - The design is valid only if it prevents:
    - a new lane for every tangent
    - a visible chat-thread maze
    - internal orchestration lanes masquerading as user topic lanes
    - user topic lanes silently conflicting with system lane concepts

PHASE 7A: Sidecar Lifecycle Recovery Lock (MANDATORY, APP-OWNED)
- Root-cause framing:
  - Sidecar-down recovery is an actuator / permissions / ownership gap.
  - It is not a worker-lane self-improvement problem.
  - Worker lanes may not redesign themselves, expand their authority, or rewrite governance because they detected this gap.
- Final recommended architecture:
  - xMilo app owns sidecar lifecycle authority.
  - Restart authority must be narrow, explicit, and allowlisted.
  - Normal recovery must not depend on ad hoc ADB or manual Termux interaction.
  - If Termux remains the sidecar host in v1, the app may invoke only one approved restart entrypoint for sidecar recovery.
  - No arbitrary shell execution and no broad Termux control surface are permitted.
- V1 decision:
  - Use a single owned restart entrypoint / service action.
  - If a bridge into Termux is required, it must expose only that single approved restart action and nothing broader.
- Ownership lock:
  - Main App Builder lane owns the lifecycle controller, restart trigger, restart-state UX, and recovery verification surface.
  - Milo Mind lane owns only sidecar health/ready/auth semantics and must not own restart authority.
  - Main Hub owns the governance, allowlist, and operator-fallback template.
- Restart trigger contract:
  - App may trigger restart when sidecar-down state is detected by the approved health checks or when the approved recovery policy determines the sidecar is unavailable or unhealthy.
  - Approved restart target is only the owned sidecar restart entrypoint using the canonical config path.
  - App must not send arbitrary command text, arbitrary file paths, or arbitrary arguments through the bridge.
- Recovery-proof contract:
  - Do not mark restart successful unless all required checks pass:
    - `GET /health`
    - `GET /ready`
    - auth state verification when the current flow depends on auth/entitlement
  - Recovery proof must be tied to the same restart attempt and recorded as one bounded result:
    - `restart_verified`
    - `restart_failed`
    - `restart_rate_limited`
    - `restart_needs_operator`
- Repeated-failure handling rule:
  - Use bounded retry with rate limit/backoff.
  - V1 ceiling:
    - attempt 1: immediate
    - attempt 2: after short cooldown
    - attempt 3: after longer cooldown
  - If verification still fails after the bounded retry budget, stop autonomous restart and switch to the single approved operator fallback.
  - Do not loop indefinitely.
  - Reset the retry budget only after a fully verified healthy state.
- Lane behavior rule:
  - Worker lanes may detect sidecar-down state.
  - Worker lanes may report proof and request app-owned restart.
  - Worker lanes may not invent new Termux commands.

PHASE 18A: Weekly Runtime Integrity Check (PHASE 1, MANDATORY, READ-ONLY, NO DRIFT)

- Phase purpose:
  - Implement a narrow, low-risk, policy-backed weekly runtime integrity verification feature for xMiloClaw.
  - This phase is not a freeform filesystem intelligence feature.
  - This phase must not perform broad anomaly hunting.
  - This phase must remain inside approved xMiloClaw-managed runtime roots and approved Phase 1 finding semantics only.

- Absolute Phase 1 scope lock:
  - Scan only explicit xMiloClaw-owned managed runtime roots that are marked Phase 1-ready in the canonical managed-root registry.
  - Validate only against explicit registry, policy, manifest, database, or reference rules that are declared before the run begins.
  - Detect only approved Phase 1 finding classes.
  - Report calmly and non-destructively.
  - Support `some items need review` as a valid outcome.
  - Remain read-only for the full phase.

- Absolute Phase 1 finding-class lock:
  - Allowed finding classes only:
    - `possible_orphaned_artifact`
    - `possible_duplicate`
    - `registry_mismatch`
    - `partial_scan`
  - Prohibited / deferred classes:
    - `ghost_file`
    - `random_script`
    - `anomaly`
    - `suspicious_file`
    - `freeform_unknown_file_problem`
    - any other unapproved class
  - No lane, component, UI surface, or report may invent additional Phase 1 finding classes.

- Canonical finding contract requirement (mandatory artifact):
  - Phase 1 must define one canonical finding contract artifact/schema.
  - No implementation may proceed without it.
  - The canonical finding contract must define:
    - allowed finding classes
    - allowed outcome bands
    - allowed confidence values
    - required fields per finding
    - prohibited classes for Phase 1
    - mapping rules from finding state to user-facing outcome
    - precedence rules
    - prohibited upgrade paths
  - Allowed outcome bands only:
    - `CLEAN`
    - `REVIEW_NEEDED`
    - `CONFIRMED_ISSUE`
  - Allowed confidence values only:
    - `possible`
    - `probable`
    - `confirmed`
  - No implementation or lane may invent additional Phase 1 result classes, confidence labels, or user-facing outcome bands.

- Required finding fields per finding:
  - `finding_id`
  - `finding_class`
  - `confidence`
  - `root_id`
  - `artifact_path`
  - `artifact_class`
  - `policy_snapshot_version`
  - `contract_version`
  - `evidence_summary`
  - `reference_source_type`
  - `reference_source_id` if applicable
  - `grace_state`
  - `exception_state`
  - `requires_recheck`
  - `user_outcome_band`
  - `confirmed_gate_passed`
  - `created_at`
  - `updated_at`
  - `run_id`
  - `stable_for_run`
  - `notes` (compact, non-speculative)
  - Missing optional metadata may not be treated as proof of an issue unless policy explicitly requires that metadata for the root/class.

- Managed-root registry requirement (mandatory control artifact):
  - Phase 1 must define one canonical managed-root registry artifact.
  - No directory may be scanned unless it appears in that registry and is marked Phase 1-ready.
  - Each managed-root registry entry must define all of the following:
    - `root_id`
    - exact root scope / path family
    - owner component / owner lane
    - allowed file classes
    - whether duplicates are ever valid there
    - canonical reference source
    - grace-period rules
    - exception rules
    - whether the root is Phase 1-ready
    - performance budget notes if relevant
    - exclusion state for the run
  - Root readiness rule:
    - A root may not participate in Phase 1 unless all required policy fields are defined.
    - If any required policy input is missing, exclude that root from scanning rather than guessing.
  - No scanning outside registered Phase 1-ready managed roots.

- Managed-root definition lock:
  - `managed root` means a runtime path family explicitly declared in the canonical managed-root registry, owned by xMiloClaw, with defined file classes, reference requirements, grace rules, and exception rules.
  - A path is not a managed root merely because it appears xMiloClaw-related.
  - A path is not a managed root merely because it is reachable by the runtime.

- Phase 1 in-scope roots:
  - Only xMiloClaw-owned runtime roots explicitly declared in the Phase 1-ready managed-root registry.
  - Examples may be documented for planning, but examples do not become active scope unless the registry marks them Phase 1-ready.

- Forbidden scan targets:
  - any path outside declared managed roots
  - user documents
  - gallery/media libraries
  - downloads
  - shared storage not explicitly declared as xMiloClaw-managed
  - other apps' directories
  - repo source trees
  - developer worktrees / checkouts
  - generic Termux home paths outside explicitly declared xMiloClaw-managed subtrees

- Definitions that may not drift:
  - `registry_mismatch`
    - A directly provable mismatch between a file or directory-state inside a managed root and the canonical registry / policy / required reference model for that same root and artifact class.
    - It is valid only when the mismatch is explicitly checkable from declared policy.
  - `possible_duplicate`
    - A duplicate candidate inside Phase 1-managed scope whose equivalence is supported by allowed duplicate-policy evidence but has not met confirmed threshold for that run.
  - `possible_orphaned_artifact`
    - A managed-root artifact whose required owning reference is missing or unresolved under declared policy, but the evidence has not passed confirmed threshold for that run.
  - `partial_scan`
    - A scan-completeness state indicating that one or more required checks for the run did not finish due to runtime incompleteness or execution limits.
    - `partial_scan` is not, by itself, proof of a confirmed integrity issue.
  - `confirmed issue`
    - A finding that passes the full confirmed gate using only allowed Phase 1 classes, stable evidence, and policy-backed validation.
  - `review-needed`
    - A calm, non-dramatic user-facing state meaning the scan completed with one or more items requiring review or recheck, but no confirmed critical issue was detected.
  - `confidence`
    - The contract-defined evidence-strength label for a finding: `possible`, `probable`, or `confirmed`.
  - `grace period`
    - A declared policy window during which an artifact may exist, lack a final reference, or remain transient without being promoted to a confirmed finding.
  - `exception rule`
    - A declared policy rule that protects a root, file class, or artifact state from normal classification or changes the classification threshold.
  - `validly skipped by policy`
    - A root or check omitted because the policy snapshot declared it not required before the run began.
    - Valid skip examples:
      - root intentionally disabled for this build/profile
      - root intentionally excluded by declared Phase 1 policy
      - root intentionally deferred by declared schedule/policy rule
    - Not valid policy skips:
      - permission unavailable
      - root inaccessible unexpectedly
      - runtime budget exhausted
      - battery/thermal deferral
      - transient filesystem/access error
      - scanner interruption
      - any other execution-time incompleteness

- Anti-drift evidence rules:
  - No file may be classified from filename alone.
  - No file may be classified from extension alone.
  - No file may be classified from `looks suspicious` reasoning.
  - Unfamiliar file appearance is not evidence.
  - Missing metadata alone is not proof of a problem unless policy explicitly requires that metadata/reference for the root/class in question.
  - Unknown must stay unknown.
  - Absence of certainty must remain uncertainty and must not be upgraded into suspicion.
  - If policy/reference support is insufficient, the result must remain soft, deferred, excluded, or review-needed.
  - No confirmed issue may exist without policy-backed validation.

- Deferred-category firewall (mandatory):
  - Deferred concepts must not leak into Phase 1 runtime behavior, outputs, logs intended for review, or user messaging.
  - Deferred concepts include:
    - ghost files
    - random scripts
    - anomaly hunting
    - freeform weirdness detection
    - anything outside approved Phase 1 classes
  - No weaker lane, UI surface, or implementation may translate a Phase 1 finding into a deferred concept.

- Non-destructive enforcement:
  - Phase 1 is explicitly read-only.
  - No delete behavior is permitted.
  - No move behavior is permitted.
  - No quarantine behavior is permitted.
  - No rewrite behavior is permitted.
  - No cleanup behavior is permitted.
  - No silent remediation is permitted.
  - No auto-fix interpretation is permitted.
  - Phase 1 is detection, classification, persistence, and calm reporting only.

- Policy snapshot and run-freeze rules:
  - Every run must evaluate against one frozen policy snapshot for that run.
  - The policy snapshot must include, at minimum:
    - Phase 1-ready managed roots
    - allowed file classes
    - exception rules
    - grace rules
    - root readiness / exclusion state
    - canonical reference requirements
    - finding contract version
    - outcome precedence rules
  - No mid-run policy drift:
    - policy, registry, or readiness changes after run start must not retroactively change classification inside that same run
  - Required-root set freeze:
    - the set of required roots for a run must be fixed at run start from the policy snapshot
    - a root may not become `not required` midway through the run due to later state changes

- Normalization rules:
  - Before classification, normalize path representation according to platform rules.
  - Normalize path separators.
  - Normalize equivalent path references.
  - Guard against relative/absolute path duplicates.
  - Guard against aliasing or multi-reference representations of the same artifact when relevant.
  - Representation differences alone must not create duplicates or mismatches.

- Runtime-completeness and mutation rules:
  - If runtime incompleteness affects required coverage, `CLEAN` must not be returned.
  - If a file, registry reference, or required artifact changes during the scan window in a way that makes classification unstable or non-reproducible, do not promote it to `CONFIRMED_ISSUE` from that run alone.
  - If an item appears, disappears, changes size/hash/metadata, or gains/loses its reference during the scan window, classify conservatively.
  - Transient-write protection:
    - Files currently being created, rewritten, compacted, exported, migrated, or otherwise actively mutated must not be treated as confirmed duplicates/orphans/mismatches unless policy explicitly defines a stable validation method for that lifecycle state.
  - Recheck rule:
    - where practical, unstable findings must prefer later recheck over confirmation in the current run.

- Two-pass confirmation rule:
  - For non-urgent findings, use two-pass confirmation where practical.
  - First pass may produce `possible` or `probable` review-level findings.
  - Second pass may confirm only if the issue remains after grace and revalidation.
  - Two-pass confirmation is required to reduce false positives from transient lifecycle states.

- Outcome precedence and gating:
  - Deterministic precedence:
    - 1) If one or more policy-backed confirmed findings exist, outcome = `CONFIRMED_ISSUE`
    - 2) Else if one or more review-level findings exist, outcome = `REVIEW_NEEDED`
    - 3) Else if scan completed within approved scope and no findings exist, outcome = `CLEAN`
    - 4) Else if the only finding is `partial_scan`, outcome = `REVIEW_NEEDED`
  - `partial_scan` gate:
    - `partial_scan` is a scan-completeness state, not a confirmed integrity issue
    - `partial_scan` alone must never produce `CONFIRMED_ISSUE`
    - `partial_scan` may block `CLEAN`
    - `partial_scan` may produce `REVIEW_NEEDED`
  - confirmed-issue gate:
    - A finding may be treated as confirmed only if all are true:
      - file is in a managed Phase 1 root
      - root policy is defined
      - file class is defined
      - required registry/reference model exists
      - required reference/ownership validation fails
      - grace period has elapsed
      - no exception rule applies
      - confidence threshold for confirmed is met
      - finding class is allowed in Phase 1
      - result is not based solely on incomplete scan, missing metadata, filename shape, extension, or heuristic suspicion
      - the evidence used for classification is stable for that run
  - review-needed gate:
    - `REVIEW_NEEDED` may be used only for:
      - `partial_scan`
      - `possible_orphaned_artifact`
      - `possible_duplicate`
      - `registry_mismatch` below confirmed threshold
      - bounded duplicate candidate awaiting recheck
      - unstable findings requiring recheck
    - `REVIEW_NEEDED` must not be described as confirmed breakage
  - `CLEAN` gate:
    - `CLEAN` may be returned only when:
      - every required Phase 1 root was either fully checked or explicitly excluded by declared policy before the run began
      - no review-worthy candidate remains
      - no confirmed issue exists
      - no runtime incompleteness affected required coverage
